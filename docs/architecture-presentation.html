UltraTrader 2026 · Architecture Redesign

Five-Component
Program Architecture

A clean separation of concerns — from monolithic strategy configs to modular, reusable, composable trading programs.

The Problem

Everything is in one blob

Current StrategyConfig
Entry conditions + Stop + Target
+ Timeframe + Session windows
+ PDT rules + Gap risk
+ Position sizing + Leverage
+ Risk caps + Regime filter
+ Cooldown rules + Order type
+ Slippage model + Commission
= 60+ fields, impossible to reuse
The Result
Every strategy is unique. Nothing is reusable. Creating a new strategy means filling out the same session hours, sizing rules, and order mechanics from scratch every time.
The Vision
A Strategy is just a signal.

"Fast EMA crosses above slow EMA → go long. Stop at swing low. Target at 2R."

Everything else — when to fire, how to order, how much to risk, which symbols — lives in separate reusable components that snap into a Program.
The Result
Build a strategy once. Attach different governors, execution styles, risk profiles, and watchlists to create unlimited programs. Backtest the program, not the strategy.

Component 1

Strategy — Pure Signal Logic

What it contains
  • Entry conditions (indicators, price action)
  • Stop definition (swing low, ATR multiple, fixed %)
  • Target definition (R-multiple, ATR, resistance)
  • Logical exit conditions (exit when RSI does X)
  • Hypothesis — the edge in plain language
What it does NOT contain
  • ❌ Timeframe
  • ❌ Session windows or PDT rules
  • ❌ Position sizing or leverage
  • ❌ Risk caps or portfolio limits
  • ❌ Alpaca order type or bracket logic
  • ❌ Watchlist or symbols
Result
A strategy is portable. One EMA crossover strategy can power a day-trading program on 5m bars AND a swing program on daily bars. No duplication.

Strategy Builder form → 3 tabs only: Core (identity) · Signals (entry/exit conditions) · Advanced

Component 2

Strategy Governor — When to Fire

What it controls
  • Timeframe (5m, 1h, 1d, etc.)
  • Duration mode (day / swing / position)
  • Session windows — only fire 9:35–11:00 ET
  • Force-flat by time (e.g. 15:45 close)
  • Regime filter — only in trending_up
  • Cooldown rules — wait 60 min after stop-out
  • Max trades per session cap
  • Earnings blackout — close before earnings
  • PDT protection rules
Golden templates
Day Trade NYSE 5m
5m · day · 9:35–11:00 + 13:30–15:00 · flat by 15:45

Swing Daily 1d
1d · swing · no session restrictions

Position Weekly 1d
1d · position · earnings blackout · weekend allowed
Why separate?
An ORB strategy at 2pm is not a valid ORB signal. The time filter IS part of signal validity — but it's reusable. The same governor can be attached to multiple strategies.
Component 3

Execution Style — Alpaca Order Mechanics

The key insight
The strategy says "entry here, stop here, target here."

The Execution Style answers: HOW does Alpaca actually place that?

It reads the strategy's prices and translates them into Alpaca's native order types.
Fields
Entry order type: market / limit / stop / stop_limit
Time-in-force: day / gtc / ioc / opg
Bracket mode: bracket / OCO / trailing_stop / none
Limit offset method: ATR × N or % pullback
Cancel after N bars if unfilled
Scale-out: close 50% at T1, trail the rest
Golden templates
Bracket Market Entry
Enter market → auto-attach OCO bracket (TP + SL linked)

Limit Pullback Entry
Enter limit at close − 0.5×ATR · cancel after 3 bars

Stop-Limit Breakout
Trigger at breakout level, fill at limit or better

Trailing Stop Exit
Enter market · trail exit at 2% below highest close
Backtest assumptions
Fill model (next_open / bar_close) · Slippage BPS · Commission per share — absorbed from old execution_policy blob.
Component 4

Risk Profile — Sizing + Caps

What it contains
  • Max risk % per trade — this implies position sizing
  • Shares = dollar_risk ÷ (entry − stop)
  • Max concurrent positions (long + short separately)
  • Max daily loss %
  • Max portfolio heat %
  • Max drawdown lockout %
  • Max correlated exposure %
  • Max leverage
Two attachment points
Program level — risk rules for this specific program while it's running.

Account Governor level — account-wide hard caps that override everything. The kill switch layer.

Same Risk Profile object. Different attachment points.
Already exists
RiskProfile model is production-ready in the backend. Only change: add risk_profile_id FK to TradingProgram so it can be attached at program level too.
Above the Programs

Account Governor

Purpose-built accounts
One account per trading style eliminates cross-style complexity:

📅 Day trading account — intraday only, PDT governed
📈 Swing trading account — 1–10 day holds
🏔️ Position trading account — multi-week, gap risk controls

Each account runs only programs matching its style.
Governor responsibilities
  • Receives signals from all programs on this account
  • Resolves conflicts: Program A buys TQQQ, Program B shorts TQQQ → Governor decides (already built: ConflictResolver)
  • Tracks positions by Alpaca client_order_id (already built)
  • Enforces account-wide kill switch
  • Controls capital allocation per program
  • Has its own Risk Profile (account-wide hard caps)
Alpaca does not allow long and short on the same symbol in one account. The Account Governor is the single point that enforces this — before anything hits Alpaca.

How it Works

Composing a Program

Step 1 — Build components
Create a Strategy: "EMA cross, stop at swing low, target 2R"

Create a Governor: "Day Trade NYSE 5m — fire 9:35–11:00 only"

Create an Exec Style: "Bracket Market Entry — enter market, auto-attach OCO bracket"

Pick a Risk Profile: "Conservative — 1% risk per trade, max 5 positions"

Pick a Watchlist: "S&P 500 High Momentum"
Step 2 — Compose the Program
Snap all five together in the Program builder. Each slot shows a picker modal. Programs are frozen before deployment — all component refs locked.
Step 3 — Backtest the Program
The backtest engine resolves all components into one flat config dict — same format it's always consumed. No breaking changes.
Step 4 — Deploy to Account
Frozen program deployed to purpose-built account. Account Governor receives all signals, resolves conflicts, enforces hard caps.
Backtesting

Strategy config (base) + Governor fields (timeframe, windows, regime) + Exec fields (entry_module) + Risk fields (sizing, caps) + Watchlist symbols = one flat dict the engine already knows how to read
Backward compatible
BacktestLauncher keeps its "Strategy Version" mode. Existing saved strategies continue to work unchanged. New path: select a frozen Program instead.
What backtest recommends
Walk-forward and optimization produce program recommendations — best Governor config, best Risk Profile settings. "Generate Program" button creates a draft from the results.
